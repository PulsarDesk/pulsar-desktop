### C1 [high] RK3588 reopen: a codec/monitor switch whose new stream fails decoder validation kills the decode thread permanently (retries once, then dies — black for the rest of the session, no recovery)
- **REGRESSION** | platforms: linux | dim: regress-render | files: desktop-app/crates/pulsar-render/src/video.rs:231, desktop-app/crates/pulsar-render/src/video.rs:253, desktop-app/crates/pulsar-render/src/video.rs:260, desktop-app/crates/pulsar-render/src/video.rs:277, desktop-app/crates/pulsar-render/src/video.rs:462
- scenario: On an Orange Pi / RK3588 (aarch64) client in an active session, the user switches the codec (or switches the host monitor, which reuses the codec-switch reopen path) to a codec whose decoder fails the Moonlight-style one-frame validation probe — e.g. AV1, which has no rkmpp HW decoder and whose software fallback can fail to validate on RK3588 (decode::select() returns None), or any codec whose decoder disappears/validation fails. This is the documented 'av1 fails -> degrade' class.
- symptom: The video goes permanently black/frozen for the rest of the session with no recovery. The pulsar-render process stays alive but its decode thread has exited, so even switching BACK to a known-good codec (H.264/H.265) does nothing — the renderer is dead until full disconnect/reconnect.
- root cause: request_reopen() sets REOPEN_SDP+REOPEN(=true)+SWITCHING(=true). The running decode_once exits on REOPEN; start_decode (video.rs:242-263) takes the new SDP, clears REOPEN, and runs decode_once on it. If that pass hits `break 'decode` (e.g. decode::select() -> None at video.rs:462-467, 'no decoder validated') before any keyframe, decode_once returns with SWITCHING still true. The just-landed recovery branch `if next.is_none() && SWITCHING` (video.rs:253-263) clears SWITCHING, sleeps 500 ms, and requeues the SAME failing SDP exactly once. The retried pass `break 'decode`s again; this time `next.is_none()` is true but SWITCHING is now false, so the branch is skipped and start_decode hits `None => return` (video.rs:277), terminating the decode thread. No code path ever re-spawns the decode thread — request_reopen only flips atomics, and nothing consumes them once the thread is gone. So a single failed-validation reopen is a permanent kill (the renderer must survive on RK3588, but its decoder doesn't).
- suggested fix: Distinguish a transient open/timeout failure (retry) from a hard, non-recoverable decoder-validation failure, and do NOT let start_decode's loop fall through to `return` while the renderer process is meant to persist. Simplest: in the `next.is_none() && SWITCHING` recovery branch, keep retrying THIS sdp on a backoff (do not clear SWITCHING permanently after one attempt) OR, on a validation `break 'decode`, leave REOPEN_SDP=Some(sdp) and keep the thread parked in a poll loop that re-arms on the next request_reopen, so a subsequent switch back to a decodable codec revives video. At minimum, never terminate the decode thread on RK3588 — loop with a sleep waiting for the next REOPEN_SDP instead of returning.
- verifier correction: The mechanical defect is real and confirmed in the just-landed code (commit d2b4048): a reopen pass that ends via any pre-read-loop `break 'decode` (video.rs:453/466/472/492/502) or the early `return` after avformat_open_input retries fail (video.rs:433-435) leaves REOPEN_SDP=None with SWITCHING=true; the recovery branch (video.rs:253-263) clears SWITCHING, sleeps 500ms, retries the SAME sdp exactly once, and if that second pass fails the same way, control falls to `None => return` (video.rs:277) and the decode thread terminates. start_decode is the only decode-thread spawn (video.rs:229, called once at linux.rs:493); request_reopen (video.rs:81, from the `reopen` stdin line at linux.rs:402) only flips atomics that nothing consumes once the thread is gone, so there is no revival path — switching back to a good codec does nothing. That matches the symptom and contradicts the renderer's own "this process must survive" intent (linux.rs:399).\n\nHOWEVER the bug's HEADLINE causal story is inaccurate/refuted: the specific 'switch to AV1 whose software fallback fails validation -> select() returns None' path is unreachable through normal negotiation. The client advertises only probe-validated codecs (play.rs:182 filters d.ok), the negotiated decode_codecs/allowed set = host-encodable ∩ client-decodable (play.rs:201,741), and the host clamps every stream AND user codec-switch (Restream::Codec) to req.decode_codecs (handlers.rs:1511-1521; hold.rs:528/588 always carry decode_codecs). A codec whose select() returns None at startup is ok:false, never in decode_codecs, and can never be put on the wire. Tier-2 software is always a candidate (decode.rs:188-197) and deterministically validates the conformant baseline, so select()->None for a negotiated codec is not the live failure mode.\n\nAccurate reachable trigger: any reopen (codec OR monitor switch share the path) whose NEW stream transiently fails to open/init TWICE within ~500ms — avcodec_open2 failure (video.rs:500-502), av_hwdevice_ctx_create failure (rkmpp/DRM device busy during the host restream gap, video.rs:489-492), avformat_open_input failing after retries (video.rs:433-435), or a temp-dir write failure during re-validation. The just-landed fix converted 'permanent death on the FIRST failed-validation reopen' into 'death after the SECOND' — it bounds but does not eliminate the permanent kill. The suggested fix is directionally correct: do not let start_decode's loop `return` while the renderer must persist; park in a poll loop that re-arms on the next REOPEN_SDP, and/or keep retrying THIS sdp on backoff instead of clearing SWITCHING permanently after one attempt.
- **FIX [fixed]:** The decode thread in start_decode could die permanently after a failed-validation reopen (e.g. AV1 on RK3588 where decode::select() returns None). The old recovery branch cleared SWITCHING and retried the same SDP exactly once; when the retry also broke 'decode, SWITCHING was now false so the branch was skipped and `None => return` terminated the thread. The fix removes the one-shot retry and instead parks the thread in a 200ms poll loop on REOPEN_SDP when it would otherwise return. When a new reopen arrives (e.g. switching back to H.264), the thread takes the new SDP, clears REOPEN, and re-enters the decode loop — preserving the thread across any number of failed-validation reopens for the lifetime of the renderer process.
  - follow-up: The park loop polls every 200ms while parked after a failed validation. This is low overhead but could be refined to use a Condvar if desired. Also: the session-level code that calls request_reopen may want to emit a user-visible "codec not supported, switch back" error message when SWITCHING goes false without a keyframe ever appearing — currently the overlay just stops showing the spinner with no explanation.

### C2 [high] go_online reconnect leaks the old Node (and pins its UDP socket) because the media-forwarder tasks (vh/ah) are never aborted on abort() — defeats the C17/C3 "await old socket freed before rebind" fix
- **REGRESSION** | platforms: all | dim: teardown-reconnect-concurrency | files: desktop-app/src-tauri/src/host.rs:66, desktop-app/src-tauri/src/host.rs:144, desktop-app/src-tauri/src/host.rs:1075, desktop-app/src-tauri/src/host.rs:1095, desktop-app/src-tauri/src/host/handlers.rs:918
- scenario: A client is connected to this host over a media-over-session (FEAT_MOS) stream — the normal relay-fallback / symmetric-NAT path, so essentially every relayed session. The host user changes the relay endpoint or network mode in Settings, or hits retry / go online again, which re-runs go_online while the session is live.
- symptom: After the reconnect, the host cannot rebind a pinned cfg.node_port (intermittent address-in-use → go_online returns err.portInUse), and the old Node's relay heartbeat keeps pinging the OLD relay. The old session's two RTP forwarder tasks keep running idle for the rest of the app's life (each blocked in vsock/asock.recv()), pinning the old Arc<Node> at strong-count > 0 so its UDP socket is never released. This is the exact failure C17 claimed to fix.
- root cause: go_online (host.rs:144) aborts + awaits only state.session_tasks and serve_task. The B25/C17/C3 fix added a SessionCleanupGuard (host.rs:66) whose Drop kills ffmpeg procs and closes the Linux portal on abort, but it does NOT touch fwd_slot. The vh/ah media-forwarder tasks are independent tokio::spawn()s (handlers.rs:918,990) tracked only in the per-session-local fwd_slot (extended at handlers.rs:1002) and aborted ONLY in the post-tokio::select! teardown block (host.rs:1095) — a path that JoinHandle::abort() of the session task skips entirely. Each forwarder holds vtx/atx = media_tx.clone() = SessionSender { node: Arc<Node> } (session.rs:109), a strong clone. After the guard kills the capture/ffmpeg, the loopback intake socket goes quiet so the forwarder blocks forever in recv() (it never errors and vtx.send() never fails because the send itself keeps the Node alive), so the old Node never reaches strong-count 0 and its socket stays bound. The C17 verifier explicitly called out this exact forwarder gap; the applied fix awaited the session tasks but left the forwarders unreachable.
- suggested fix: Make the forwarder tasks reachable from go_online's deterministic teardown: either move the fwd_slot abort into SessionCleanupGuard::Drop (drain+abort vh/ah alongside the procs kill), or track the forwarder JoinHandles centrally so go_online's drain loop aborts+awaits them too. The forwarder loops should also exit on session loss — e.g. select! against a shutdown notify — so they don't idle-pin the Node.
- verifier correction: The claim is correct in conclusion and root cause but overstates that BOTH forwarders leak. The VIDEO forwarder (vh, handlers.rs:918) does self-terminate: its loop selects on `nack_rx.recv()` too, and the nack_tx sender lives only inside nack_slot, whose only Arc clones are the session-task-local (host.rs:473) and the on_nack closure captured into DataHandlers (host.rs:1040). When the session task is aborted and its future dropped, both clones drop -> nack_rx closes -> `q = nack_rx.recv()` yields None -> `let Some(seqs) = q else { break }` breaks vh, releasing its SessionSender. So vh's Arc<Node> is released (asynchronously, after go_online has already awaited the session task — so it may still be alive at the rebind instant, a transient race, but it does eventually drop). The AUDIO forwarder (ah, handlers.rs:990) has NO such escape: `while let Ok(n) = asock.recv(&mut buf).await` with no select against any channel/shutdown. After ffmpeg is killed it parks in asock.recv() forever and never drops its atx clone. So ah alone is the permanent leak that pins Arc<Node> and the UDP socket. Fix: either abort+drain fwd_slot inside SessionCleanupGuard::Drop (or track the forwarder JoinHandles centrally so go_online aborts+awaits them), AND give ah a shutdown escape (select against a notify, mirroring vh's nack_rx behavior) so it can exit on its own.
- **FIX [fixed]:** Added fwd_slot to SessionCleanupGuard so that when go_online aborts session tasks (via JoinHandle::abort()), the Drop impl also aborts the vh/ah RTP media-forwarder tasks. Previously those tasks held strong Arc<Node> clones (via SessionSender) and blocked indefinitely in vsock/asock.recv(), keeping the old Node and its UDP socket alive and bound — defeating the C17/C3 port-rebind fix. The abort is non-blocking (same pattern as the procs kill: no await in Drop to avoid stalling the executor thread). The normal post-select! teardown path already drained fwd_slot, so the guard's drain is a safe no-op on that path.
  - follow-up: The forwarder loops could additionally exit early on session loss by selecting against a shutdown notify/channel, giving a belt-and-suspenders defense. However the guard abort is the minimal correct deterministic fix and is sufficient on its own.

### C4 [high] Successful mid-session monitor switch leaves absolute (webview) pointer input mapped to the PREVIOUS monitor — permanently, in the common case
- **REGRESSION** | platforms: windows | dim: display-monitor | files: desktop-app/src-tauri/src/host/handlers.rs:1121, desktop-app/src-tauri/src/host/handlers.rs:1128, desktop-app/src-tauri/src/host/handlers.rs:1129, desktop-app/src-tauri/src/host/handlers.rs:1137, desktop-app/src-tauri/src/host.rs:744
- scenario: Windows multi-monitor host, native NVENC path, live REMOTE session with a webview/absolute-pointer client (Windows->Windows webview, or any client sending InputEvent::PointerMotion). The user picks monitor B from the session menu while currently streaming monitor A. The client sends exactly ONE StreamReq{display_idx=B}; the switch SUCCEEDS. The user then keeps working on B without changing any other stream setting (no further restream).
- symptom: The video correctly moves to monitor B, but every absolute pointer move/click lands on monitor A (the previously-streamed screen) — the cursor and clicks land on a screen the user is no longer viewing. This persists for the entire rest of the session (or until an unrelated restream, e.g. a codec/res/fps change, happens to fire on_stream again). Non-primary monitor control after a switch is effectively broken.
- root cause: In make_on_stream's fast path, after issuing h.switch_output(req.display_idx) (handlers.rs:1121) the code IMMEDIATELY reads `let actual = h.current_output()` (handlers.rs:1128) and stores it into cur_display (1129) and last_native_req.display_idx (1137). But the capture thread has not rebuilt yet, so current_output() still returns the OLD output A. cur_display is therefore set to A even though the switch will succeed and land on B. The on_input closure reads `cur_display.load()` (host.rs:744) and only re-resolves display_rect()/set_monitor() when the index CHANGES (`if idx != applied_display`). Because cur_display stays A and there is no subsequent on_stream call after a one-shot menu pick, applied_display never updates, so the DesktopInput's MonitorRect stays pinned to A and pointer() maps the normalized 0..1 coords onto monitor A's virtual-desktop rect forever. The capture thread does write current_output=B after its rebuild, but NOTHING in the input path reads h.current_output() live — only cur_display, which is only written inside make_on_stream. The C1 fix in BUG-HUNT-2 correctly handled the FAILED-switch revert (so input doesn't map to an unreachable B) and its own follow-up dismissed the successful-switch case as a 'minor transient latency [that] self-corrects on the next on_stream call' — but for a one-shot menu switch there is frequently NO next on_stream call, so the wrong-monitor mapping is the permanent common case, not a transient.
- suggested fix: Read the actually-confirmed output in the input path itself rather than caching it from the moment of the switch request: pass the CaptureHandle (or its current_output Arc<AtomicU32>) into the on_input closure and, in the PointerMotion arm, re-resolve set_monitor whenever h.current_output() differs from applied_display (a cheap atomic load per pointer event, or gate on a generation counter). Alternatively, after switch_output, do not store current_output() synchronously — instead have the capture thread publish a 'switch confirmed' notify and update cur_display from a small watcher, so cur_display tracks the monitor the thread actually landed on rather than the pre-rebuild value.
- verifier correction: The bug is real as described. One refinement: the wrong-monitor mapping is not unconditionally permanent — it is healed by ANY subsequent on_stream that re-enters the Windows fast path, which includes not only an explicit user restream (codec/res/fps/audio change) but also an adaptive bitrate step from hold.rs (which fires when the link shows packet loss). So it is "persistent in the clean-network common case" rather than strictly permanent. The claim already concedes this. Also note the symptom covers clicks as well as moves, because PointerButton fires at the host cursor position last set by the (mis-mapped) absolute pointer() call, so clicks land on monitor A too.
- **FIX [fixed]:** On the native DXGI+NVENC path, the on_input PointerMotion arm now reads the capture thread's own current_output Arc directly rather than cur_display. cur_display was written synchronously at switch-request time (before the capture thread rebuilt), permanently pinning the input mapping to the old monitor after a one-shot menu switch. The fix introduces native_out_arc (an Arc<Mutex<Option<Arc<AtomicU32>>>> shared between make_on_stream and on_input): the slow path stores h.current_output_arc() into it when a CaptureHandle is installed; on_input's PointerMotion arm reads from this Arc when present (native path), falling back to cur_display for the ffmpeg path. Because the capture thread writes current_output only after each confirmed build including reverts, on_input always resolves the set_monitor call against the monitor actually being streamed. The C1 revert-protection is preserved: a failed switch causes the thread to revert to the previous monitor and write that value to current_output, so the input mapping correctly reverts too.
  - follow-up: The cur_display.store(actual, ...) call in the fast path (handlers.rs line 1133) is now redundant for the native path since on_input reads from native_out_arc instead. It remains harmless (cur_display is still used as the fallback for the ffmpeg path) but could be cleaned up in a later pass.

Compiles clean (cargo check -p pulsar-tauri: 0 errors, only pre-existing warnings).

### C6 [high] Linux disconnect now SIGTERM-kills the native renderer every time — directly contradicts the documented 'destroying the renderer EGL context corrupts WebKitGTK's shared Mali GL → reboot needed' root cause that the resident-renderer design existed to avoid
- **REGRESSION** | platforms: linux | dim: process-lifecycle | files: desktop-app/src-tauri/src/play.rs:1007, desktop-app/src-tauri/src/play.rs:1018, desktop-app/src-tauri/src/play.rs:949, desktop-app/src-tauri/src/play.rs:961, desktop-app/src-tauri/src/play.rs:1044
- scenario: On an RK3588 kiosk, a session ends (host kicks / link drop → play-ended → stop_stream, or the user disconnects). The B7/B16 fix replaced the old 'write hide + drop resident' teardown with stop_render_child (SIGTERM, 600 ms grace, then SIGKILL) on the embedded `--wid` pulsar-render, then destroys the container.
- symptom: Risk of the documented WebKitGTK GL-wedge returning: after the session ends the home/webview renders but stops processing clicks (hover works, nothing is clickable) with no in-session recovery short of reboot — the exact failure the prior resident-renderer model was built to prevent. If it occurs it happens on EVERY disconnect, not just abnormal exit; an auto-reconnect kiosk would wedge its own shell.
- root cause: The original diagnosis preserved in the OLD code comment and in MEMORY stated plainly that 'Killing destroys its EGL context, which corrupts the WebKitGTK webview's shared Mali GL on RK3588 and wedges the webview (clicks dead) with no in-session recovery (needs a reboot)' — which is why stop_stream used to send `hide` and DROP the child (keep it resident) instead of killing it. The new code (play.rs:1018) unconditionally calls stop_render_child, and the new comment (play.rs:1007-1017) asserts that an orderly SIGTERM teardown (release EGL context, XDestroyWindow, XCloseDisplay) is safe and 'only a hard SIGKILL mid-EGL-bind would wedge it'. That re-classification reverses the documented root cause without a verified basis: if EGL-context DESTRUCTION on the shared Mali display is itself the corrupting event (as the prior diagnosis says), a clean SIGTERM teardown destroys the very same context and re-introduces the wedge. Additionally, stop_render_child's 600 ms grace falls back to SIGKILL (play.rs:961) — if the renderer is busy on present/decode and doesn't reach STOP within 600 ms, it IS hard-killed mid-EGL, the case both old and new comments agree is fatal.
- suggested fix: Re-verify on RK3588 hardware that a clean SIGTERM teardown of the embedded renderer does NOT wedge WebKitGTK before relying on it. If it does, restore a safe model: keep ONE resident renderer per app and REUSE it on reconnect (send `show` + a fresh SDP over its existing stdin) so no EGL context is ever destroyed mid-app-life, while reaping all-but-the-latest hidden renderer to avoid the orphan pile-up B7/B16 targeted.
