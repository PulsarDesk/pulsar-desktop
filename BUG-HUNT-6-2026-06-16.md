### C1 [high|REG] Mid-session host display ROTATION change is never picked up on the native DXGI+NVENC path → the stream bakes the stale orientation (sideways/upside-down video) for the rest of the session
- files: pacing.rs:147, pacing.rs:152, lib.rs:312, encoder.rs:76, submit.rs:228 | symptom: The video keeps flowing but is now rendered with the WRONG orientation: the client shows the (now-portrait) desktop inside an unchanged landscape frame, sideways/squished, for the entire rest of the session. - **FIX [fixed]:** Added `built_rotation` baseline (captured from `self.rotation_deg()` before the pacing loop, same as `built_w`/`built_h`) and extended the Reinit-arm `if` condition in `pacing.rs` to also return `RunExit::Switch(self.output_idx)` when `self.rotation_deg() != built_rotation`. Compiles clean (`cargo check -p pulsar-capture`).

- files: handlers.rs:197, node.rs:217, lib.rs:384, lib.rs:392, lib.rs:333 | symptom: The relay replies ErrCode::Protocol/"no such session" (relay/src/lib.rs:384; or "not a session member" :392; Accept paths :333/:336). ### C4 [high|REG] Resident-renderer pool (AppState::resident_render Vec) grows unbounded to the high-water mark of concurrent tabs and is never reaped — reintroduces the documented RK3588 orphan-renderer pile-up / stutter
- files: play.rs:478, play.rs:1330, play.rs:827, play.rs:981, state.rs:121 | symptom: After any episode of N concurrent tabs, N pulsar-render processes stay resident forever (until full app exit). The pool only ever shrinks by one per connect (`.pop()`), so once usage returns to a single tab the pool settles at N and never drains below it. Each leaked renderer keeps an alive EGL/GL context, an X child window, a running decode thread, and a bound RTP UDP socket, all contending for the Mali GPU/CPU — the streamed video sags below panel rate with periodic ~100 ms hitches and the host looks 'stale' (the exact opi5-input-stutter / orphan-pile-up class). PR_SET_PDEATHSIG only reaps them on app exit, not between sessions.
- **FIX [fixed]:** Added `reap_excess_resident_pool` helper (Linux-only, `#[cfg(all(unix, not(target_os = "macos")))]`) that drains and SIGTERM-kills all-but-one parked resident from `AppState::resident_render` after each push. Called at all three push sites (request_launch failure re-park at play.rs ~835, request_stream failure re-park at ~991, and normal teardown park at ~1385). The cap of 1 prevents unbounded pool growth: after any N-tab episode, when the pool would have settled at N, all but the most-recently-parked entry are now reaped immediately. Excess parked renderers are idle (hidden window, EGL dormant) so SIGTERM causes their full clean teardown (eglDestroyContext + XDestroyWindow + XCloseDisplay) without corrupting WebKitGTK's shared Mali GL — the wedge that only occurs on SIGKILL of an actively-rendering context. Also calls `destroy_native_container` for each reaped entry so the GDK parent window is released. Comment in state.rs updated to reflect the capped-pool model. `cargo check -p pulsar-tauri` passes cleanly.

### C7 [medium|REG] The 30s download timeout silently defeats the new download concurrency guard for any transfer slower than 30s, re-tripping the very 8-transfer reassembler eviction the guard was added to prevent
- files: Files.svelte, hold.rs | symptom: Some downloads are reported 'indirilemedi' (failed) even though the network was fine: the client-side reassembler (hold.rs f_xfers, F_MAX_CONCURRENT_XFERS=8) evicts an in-progress transfer when a 9th FileBegin arrives, emitting file-recv{ok:false}. This is exactly the failure the MAX_WIRE_XFERS=6 guard was introduced to eliminate.
- **FIX [fixed]:** Fixed the 30s wall-clock timeout defeating the download concurrency guard for slow transfers. The fix converts the single-phase wall-clock timeout into a two-phase inactivity model:

Phase A (30s "no-response" timer): Guards the case where the host sends nothing at all (jailed path, network drop before FileBegin). Unchanged from before — if neither FileBegin nor file-recv arrives within 30s, the slot is released.

Phase B (120s safety backstop): Activated only when the host starts streaming (FileBegin arrives). The new `file-begin` Tauri event (emitted from hold.rs on FileBegin) triggers `onBegin()` in the pending download entry, which clears the 30s timer and arms a 120s backstop instead. This ensures slots are held for the full duration of legitimately-slow transfers (multi-hundred-MB files over relay easily exceed 30s). The 120s backstop is 2x the Rust F_XFER_IDLE_TIMEOUT (60s), giving the Rust idle sweep priority to emit file-recv{ok:false} for truly dead transfers before the frontend releases the slot as a last resort.

Changes:
- events.rs: Added FileBeginPayload struct (peer + name).
- hold.rs: Emit "file-begin" event after FileBegin is processed and the sanitized name is known.
- api.types.ts: Added FileBegin interface.
- api.events.ts: Added onFileBegin() subscription; imported FileBegin type.
- api.ts: Re-exported onFileBegin.
- Files.svelte: Import onFileBegin; add onBegin callback to each pendingDownloads entry; subscribe to file-begin and call queue[0].onBegin(); replace DOWNLOAD_TIMEOUT_MS with DOWNLOAD_NO_RESPONSE_MS (30s) + DOWNLOAD_SAFETY_TIMEOUT_MS (120s); refactor removeEntry() helper to deduplicate queue cleanup; add removeEntry() call in the safety backstop path.

### C8 [medium|new] Unauthenticated peer can pin the host's Allow/Deny popup open forever (keepalive defeats the 60s idle-deny); id-cycling spawns unlimited always-on-top, focused, attention-grabbing approval windows
- files: auth.rs:316, auth.rs:317, auth.rs:343, auth.rs:57, auth.rs:61 | symptom: The host operator's Allow/Deny window stays open indefinitely instead of auto-denying after the intended 60s of silence, because every keepalive re-arms the deadline. 2. Absolute wall-clock cap (MAX_TOTAL = 120s) added as a second tokio::pin! deadline. The popup is force-denied after 120s regardless of message activity, bounding the lifetime even for a peer that keeps submitting wrong passwords.

3. Concurrent popup cap (MAX_CONCURRENT_POPUPS = 3) enforced via a process-global AtomicUsize (OPEN_POPUP_COUNT). open_approval_window increments it on successful window creation; a new close_approval_window helper decrements it on close. race_host_auth now checks the cap before doing anything — if already at the limit it returns denied immediately with no window open, no focus steal, no attention request, breaking the id-cycling unbounded-popup DoS. No signature changes; no other call sites affected.

### C20 [low|REG] set_play_encoder's detached hide/show container task can re-show (or no-op-show) a torn-down session's container if the user disconnects during its 1200 ms sleep

### C21 [low|REG] Download completion is matched to a concurrency slot by bare filename (no transfer id), so a timed-out same-name download's late file-recv prematurely drains a different in-flight download's slot
- files: Files.svelte:54, Files.svelte:74, Files.svelte:143, Files.svelte:152, hold.rs:510 | symptom: The first download's late file-recv event shifts and resolves/rejects the SECOND download's still-pending slot entry (FIFO by name), draining the second download's concurrency slot while it is still transferring. That lets an extra transfer onto the wire, which can push the host's reassembler over MAX_CONCURRENT_XFERS=8 and silently evict a legitimate in-flight transfer — the exact failure the concurrency guard exists to prevent.
- **FIX [fixed]:** Threaded the host-assigned transfer id (`xfer`/`xferId`) through `FileBeginPayload`, `FilePayload`, the TypeScript `FileBegin`/`FileRecv` interfaces, and the frontend tracking maps. The single `pendingDownloads: Map<filename, queue>` is replaced with a two-level scheme: `waitingByName` (FIFO per filename, populated at `download()` call time) and `pendingById` (keyed by `xferId`, populated when `file-begin` arrives). `onFileBegin` promotes the head of the name queue into `pendingById[e.xferId]`; `onFileRecv` resolves/rejects by `e.xferId`. A late `file-recv` whose `xferId` is absent from `pendingById` is silently dropped — it cannot affect any other in-flight download's concurrency slot.
