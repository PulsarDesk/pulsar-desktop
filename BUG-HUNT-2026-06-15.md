### B1. make_on_stream call site missing the new last_req_store argument → Linux/macOS host fails to compile
- **Area:** host-handlers  ·  **Platforms:** linux,macos  ·  **Confidence:** high  ·  **Dims:** host-handlers-play
- **Files:** desktop-app/src-tauri/src/host/handlers.rs:839, desktop-app/src-tauri/src/host.rs:446, desktop-app/src-tauri/src/host.rs:470
- **Scenario:** Build / CI the desktop app on Linux, macOS, or ARM Linux (OrangePi/RaspberryPi) with the current uncommitted WIP. (Windows is unaffected because the new parameter is cfg-removed there.)
- **Symptom:** cargo check/build of pulsar-tauri fails on every non-Windows target — the GitHub Actions Linux/macOS/aarch64 build jobs go red and no host build is produced for those platforms. (On Windows it compiles, which is why local Windows test loops won't catch it.)
- **Root cause:** The WIP added a new final parameter to make_on_stream gated `#[cfg(not(windows))] last_req_store: Arc<Mutex<Option<StreamReq>>>` (handlers.rs:839), and the closure writes it at handlers.rs:859. But the single call site in go_online (host.rs:446) was NOT updated — the argument list ends at `cap_gen.clone()` (host.rs:470) with no last_req_store passed. On Windows the parameter is removed by cfg so arity matches; on Linux/macOS the call supplies one too few arguments → E0061 arity mismatch. host.rs's diff only added the `restream: None` field (1 line), confirming the call site was never touched.
- **Suggested fix:** Pass the new arg at the call site under the same cfg, e.g. add `#[cfg(not(windows))] last_req_store.clone(),` as the last argument to make_on_stream, and create the `Arc<Mutex<Option<StreamReq>>>` per-session in go_online (alongside the other per-session Arcs around host.rs:205-399).


## HIGH (23)

### B2. Relay restart silently changes the host's 9-digit ID; UI + LAN beacon keep advertising the dead old ID
- **Area:** connection-lifecycle/reconnect  ·  **Platforms:** all  ·  **Confidence:** high  ·  **Dims:** connection-lifecycle
- **Files:** desktop-app/crates/pulsar-core/src/connection/handlers.rs:143, desktop-app/crates/pulsar-core/src/connection/handlers.rs:83, relay/src/lib.rs:76, relay/src/lib.rs:331, desktop-app/src-tauri/src/host.rs:909
- **Scenario:** A host is online and shows its ID (e.g. 482 913 056). The relay process restarts (deploy, crash, OOM) and loses its in-memory state. The host's heartbeat (every 10s) sends Heartbeat{id,token}; the relay no longer knows that id and replies Error{NotRegistered}. handle_relay's Error arm re-sends Register. Because the relay's by_pubkey map is a plain in-memory HashMap with no persistence (relay/src/lib.rs:76, reset to empty on bind), it treats this pubkey as brand-new and mints a DIFFERENT 9-digit id, replying Registered{new_id,new_token}.
- **Symptom:** The host's RelayMsg::Registered handler (handlers.rs:83-90) overwrites self_id/token in place but emits NO event. go_online only reads the ID once (host.rs:909) and only calls discovery.set_id once (host.rs:919). So the host UI and the LAN beacon keep advertising the OLD id forever, while inner.self_id is now a new id. Anyone who tries to connect by the displayed/known ID gets TargetOffline; the host appears reachable but is not, until the user manually re-runs go_online (toggle offline/online). The maintainer's own comment at handlers.rs:145-150 incorrectly claims 'the relay maps pubkey -> id so the same 9-digit ID is reissued' — only true if by_pubkey survived, which it never does across a full relay restart.
- **Root cause:** Two compounding issues: (1) the relay's pubkey->id map is never persisted, so a restart re-issues fresh IDs (relay/src/lib.rs only ever builds an empty by_pubkey in bind()); (2) the client's re-register-in-place path updates self_id/token but never surfaces an ID change to the UI/beacon (no event from the Registered arm, set_id called only once in go_online).
- **Suggested fix:** On the client side, in the RelayMsg::Registered handler detect a changed self_id and emit a 'node-id' event + update discovery.set_id so the UI and beacon re-advertise the new ID (and warn the user the ID rotated). On the relay side, optionally persist by_pubkey to disk so an ID survives a restart, matching the documented stable-ID promise.

### B3. Unauthenticated Register flood inflates devices/by_pubkey maps; O(n) ID-minting scan amplifies it; names are unbounded
- **Area:** relay  ·  **Platforms:** all  ·  **Confidence:** high  ·  **Dims:** relay-security
- **Files:** relay/src/lib.rs:314, relay/src/lib.rs:337, relay/src/lib.rs:340, relay/src/lib.rs:35, relay/src/lib.rs:52
- **Scenario:** Attacker sends a sustained stream of ClientMsg::Register datagrams, each with a fresh random 32-byte pubkey and a large name string (the name is Option<String> with no length bound). No auth/PoW is required to register.
- **Symptom:** Memory grows: each fresh pubkey adds a Device (held for DEVICE_TTL=30s) AND a by_pubkey entry that lingers for PUBKEY_TTL = 30 DAYS even after the device is evicted, plus the attacker-supplied name String is stored verbatim. Sustained churn pins the relay's RSS for up to a month and, because every fresh registration runs an O(n) linear scan of by_pubkey for collision-checking, CPU cost per Register grows with the number of accumulated mappings — a feedback loop that degrades the whole event loop (it is single-task, so this stalls all clients).
- **Root cause:** register() at relay/src/lib.rs:314 is unauthenticated (the TODO at :320 acknowledges the map growth but defers a fix). Two amplifiers compound it: (1) the fresh-id loop at :337-344 calls self.by_pubkey.values().any(...) at :340 — a full O(n) scan of the map — for EVERY candidate on EVERY new registration, so cost is super-linear under a flood; (2) the device name (relay/src/lib.rs:52, set from the attacker-controlled ClientMsg::Register.name) is stored with no size cap, multiplying per-entry memory. PUBKEY_TTL=30 days (:35) means evicted-but-flooded mappings persist far beyond any real reconnect need.
- **Suggested fix:** Add a per-IP registration quota/rate meter (reuse Meter) before minting, cap stored name length (e.g. truncate to 64 chars), and replace the O(n) by_pubkey collision scan with a reverse index (HashSet<DeviceId> of assigned ids) so id minting is O(1). Consider a much shorter PUBKEY_TTL or an LRU cap on by_pubkey size.

### B4. File-transfer receiver buffers the whole transfer in RAM with no total-size cap (peer-driven OOM)
- **Area:** wire/file-transfer-reassembly  ·  **Platforms:** all  ·  **Confidence:** high  ·  **Dims:** wire-proto
- **Files:** desktop-app/src-tauri/src/host/handlers.rs:1818, desktop-app/src-tauri/src/host/handlers.rs:1823, desktop-app/src-tauri/src/host/handlers.rs:1828, desktop-app/src-tauri/src/play/hold.rs:328, desktop-app/src-tauri/src/play/hold.rs:333, desktop-app/src-tauri/src/play/hold.rs:338
- **Scenario:** An authenticated peer (a connected client doing a file send, or a malicious host answering an FsGet) sends a DataMsg::FileBegin then an unbounded stream of DataMsg::FileChunk frames (or chunks whose summed bytes far exceed the announced `size`/`chunks`) and never sends FileEnd. The host's make_on_file and the client's hold_session reassembly both run for any session with the file side-channel.
- **Symptom:** The receiver's memory grows without bound until the Pulsar process is OOM-killed / aborts (or the whole machine thrashes). The victim is whoever is RECEIVING the file (host on a client->host send, client on an FsGet download).
- **Root cause:** `size` (peer-controlled) only clamps the PRE-allocation (`Vec::with_capacity(size.min(64MiB))`), but the comment is explicit that `extend_from_slice` 'still grows past the clamp if a legitimately larger file actually arrives' (handlers.rs:1814-1818, hold.rs:324-328). FileChunk handling does `buf.extend_from_slice(&data)` (handlers.rs:1828 / hold.rs:338) with no running cap on total received bytes, no check of received-chunk-count against `f_expected`/`expected` during reception, and completeness is only checked at FileEnd. A peer that never sends FileEnd, or sends more than the announced size, makes the buffer grow indefinitely. Each individual chunk is bounded by the datagram/relay cap but the SUM is unbounded.
- **Suggested fix:** Enforce a hard ceiling on total reassembled bytes (e.g. a configurable max transfer size, and reject/abort the transfer the moment received bytes exceed it or exceed `size` by a margin). Also bound the in-flight transfer by `f_expected`/`chunks` (stop accepting once that many chunks arrived) and drop the buffer on a clearly-bogus FileBegin. Prefer streaming chunks to a temp file rather than holding the entire transfer in a Vec.

### B5. Cursor-shape PNG side-channel allocates output_buffer_size() from attacker-declared dimensions BEFORE the 256x256 bound check (renderer OOM)
- **Area:** wire/cursor-side-channel / native-renderer  ·  **Platforms:** windows,linux  ·  **Confidence:** high  ·  **Dims:** wire-proto
- **Files:** desktop-app/crates/pulsar-render/src/win/mod.rs:132, desktop-app/crates/pulsar-render/src/win/mod.rs:134, desktop-app/crates/pulsar-render/src/linux.rs:115, desktop-app/crates/pulsar-render/src/linux.rs:117, desktop-app/src-tauri/src/play/hold.rs:299
- **Scenario:** A malicious/compromised host sends a DataMsg::CursorShape { w: 32, h: 32, rgba_png } over the live session, where rgba_png is a tiny (few-KB) PNG whose HEADER declares a huge image (e.g. 40000x40000). hold.rs base64s it onto the renderer's stdin ('cursorimg ...'), and the native renderer (win/mod.rs / linux.rs) calls decode_cursor_png.
- **Symptom:** The separate pulsar-render process aborts/crashes (allocation failure) or the machine thrashes, killing the video for the session. A few-KB datagram triggers a multi-GB allocation.
- **Root cause:** decode_cursor_png checks the CALLER-supplied w/h (<=256) up front, but then does `let mut buf = vec![0u8; reader.output_buffer_size()];` where output_buffer_size() = output_line_size(width)*height computed from the PNG's OWN header (png-0.17 mod.rs:599), with no magnitude cap — png's Decoder `Limits.bytes` (64 MiB default) governs only the crate's internal buffers, NOT this external `vec![]`. read_info() only rejects on integer overflow of rowlen*height (checked_mul), so a 40000x40000 RGBA header passes and ~6.4 GB is allocated. The actual-vs-announced dimension check (win/mod.rs:134, linux.rs:117) happens AFTER next_frame, far too late. A giant uniform image compresses to a tiny PNG, so it fits one session datagram (direct UDP or the 16*MTU relay cap).
- **Suggested fix:** Read the PNG header dimensions (reader.info().size()) and reject if they don't equal the announced w/h (and exceed 256) BEFORE allocating the output buffer; or pass a strict png::Limits AND size-bound output_buffer_size() against w*h*4. Never size a heap allocation directly from peer-declared image dimensions.

### B6. Windows: overlay and closed-state HUD/cursor freeze during any video stall
- **Area:** windows-present  ·  **Platforms:** windows  ·  **Confidence:** high  ·  **Dims:** native-renderer
- **Files:** desktop-app/crates/pulsar-render/src/win/mod.rs:1402, desktop-app/crates/pulsar-render/src/win/mod.rs:1108, desktop-app/crates/pulsar-render/src/win/mod.rs:1224
- **Scenario:** On Windows, the user is in an active (game or remote) session and the host video stalls or pauses for a moment — e.g. they open the gaming overlay (the docs explicitly say 'while the overlay is open the video may pause/freeze and resume after'), the host window is minimized/occluded so the encoder emits nothing, the network briefly stalls, or the host's encode hiccups. While stalled, the user opens the overlay (combo) and tries to click a menu item, or just expects the stats HUD / overlay-open button / side-channel cursor to keep showing.
- **Symptom:** The egui overlay is never drawn and never responds: opening it shows only the frozen last video frame with no menu, no HUD, no overlay-open button, and no cursor; clicks/scroll appear to do nothing. It only 'wakes up' once a fresh frame finally decodes. During the very stall when the user most wants the overlay (host paused for the overlay), it is invisible/dead.
- **Root cause:** paint_overlay() is only invoked from show_frame() (on a freshly-decoded NV12 frame) and from clear(). In the main event loop, when no access units arrive (aus.is_empty()), clear() — the only path that paints the overlay while idle — is gated behind `if r.decoder.is_none()` (win/mod.rs:1403). Once the first AU has built the decoder, that branch is skipped forever, so during any stall the loop just sleeps 2ms and presents nothing, and paint_overlay never runs. wndproc keeps pushing input into EGUI_EVENTS, but those events are only drained inside paint_overlay, so input also backs up. The Linux backend does not have this bug: its render loop (linux.rs:495 loop → presenter.draw re-presents `last` + overlay::draw + swap_buffers every vsync) paints the overlay every iteration regardless of frame arrival.
- **Suggested fix:** Decouple overlay/HUD painting from fresh-frame arrival. In the idle (aus.is_empty()) branch, when the overlay is OPEN or any closed-state chrome (stats_hud/ovbtn/hint/cursor) is active, re-present the last swapchain content + run paint_overlay()/paint_closed() and Present, at a capped rate (e.g. ~60 Hz). Simplest: always call a present-with-overlay tick each loop iteration once the swapchain exists, not only when decoder.is_none().

### B7. Linux disconnect leaves the native renderer resident forever — one orphan pulsar-render accumulates per reconnect
- **Area:** linux-render  ·  **Platforms:** linux  ·  **Confidence:** high  ·  **Dims:** linux-renderer
- **Files:** desktop-app/src-tauri/src/play.rs:989, desktop-app/src-tauri/src/play.rs:1007, desktop-app/src-tauri/src/play.rs:1034, desktop-app/crates/pulsar-render/src/linux.rs:258, desktop-app/crates/pulsar-render/src/linux.rs:756
- **Scenario:** On an Orange Pi / Linux client, connect to a host, then disconnect (or the host kicks you / the link drops → play-ended → stop_stream). Reconnect. Repeat several times within one app session.
- **Symptom:** Each session leaves a hidden-but-alive pulsar-render process that never exits until the whole app quits; reconnecting spawns a brand-new one. After several reconnects multiple idle renderers pile up, each holding an EGL/GL context and a 16 ms idle loop, contending for the Mali GPU/CPU — the exact 'stutter after many reconnects / host stale' class the PDEATHSIG fix was supposed to cure (it only covers app-EXIT, not normal disconnect).
- **Root cause:** stop_stream (play.rs:989-1011) deliberately does NOT kill the Linux render_child: it writes `hide` to stdin (IDLE=true in linux.rs:258) and then DROPs the Child (play.rs:1007). Dropping a Child on Unix neither kills nor waits the process, so it keeps running resident. The design intends to REUSE that resident renderer via a `show` line, but no code path in src-tauri ever writes `show` — every reconnect calls native_view::spawn_render for a fresh play id (ids are never reused), so the resident renderer is never reactivated. Worse, immediately after the drop, destroy_native_container (play.rs:1034 → render.rs:170) removes/destroys the container GdkWindow that was the resident renderer's `--wid` parent, so the orphan is left drawing into a dead window tree (its per-frame XGetGeometry/XResizeWindow on the dead parent only survive because of x_error_ignore). PR_SET_PDEATHSIG(SIGKILL) reaps it only when pulsar-tauri itself exits.
- **Suggested fix:** Either actually reuse the resident renderer (track it per-target outside the play map and send `show` + a fresh SDP/`reopen` on reconnect instead of spawning a new one), or — since the container is destroyed anyway — drop the resident-hide model on a real disconnect and stop_render_child it (SIGTERM clean teardown) like every other path, accepting one clean GL-context teardown per disconnect. At minimum, before destroy_native_container, transfer ownership of the hidden Child so it is SIGTERM'd on the next reconnect/teardown rather than leaked for the app's lifetime.

### B8. Changing codec from the open overlay desyncs the renderer's open-state (overlay vanishes, input stays suspended) on Windows + x86 Linux
- **Area:** ui-overlay  ·  **Platforms:** windows,linux  ·  **Confidence:** high  ·  **Dims:** linux-renderer
- **Files:** desktop-app/src-tauri/src/session_cmds.rs:182, desktop-app/src-tauri/src/session_cmds.rs:263, desktop-app/crates/pulsar-render/src/win/mod.rs:204, desktop-app/crates/pulsar-render/src/linux.rs:774, desktop-app/src/lib/screens/Session.svelte:540
- **Scenario:** In game mode, open the in-session overlay (Ctrl+Shift+M), go to the Stream section, and pick a different codec (e.g. H.264 → H.265). On a non-aarch64 Linux desktop or Windows the native renderer is killed and respawned with a rewritten SDP.
- **Symptom:** The overlay visually disappears the instant the codec switch completes, but the frontend still believes the overlay is open: evdev/keyboard capture stays SUSPENDED and the container stays non-pass-through, so the user can neither control the host nor see the overlay. They appear 'stuck' until they press Ctrl+Shift+M twice (toggle off, then on).
- **Root cause:** respawn_render_for_codec (session_cmds.rs:182) kills the old renderer (which had OPEN=true) and spawns a fresh one. The fresh renderer starts with OPEN=false (linux.rs static OPEN defaults false; win/mod.rs OPEN defaults false). The re-seed block (session_cmds.rs:263-318) replays caps/ovbtn/ovbtnpos/statshud/pace/fit/audio over stdin but NEVER replays the open state (no SIGUSR1 on Linux, no `open` line on Windows). The RK3588 (aarch64) path is protected because it uses reopen_render_for_codec which keeps the same process (OPEN survives).
- **Suggested fix:** In respawn_render_for_codec, after re-seeding, if the session's overlay is currently open (state.overlay_open contains id) re-assert it: send SIGUSR1 (Linux) / writeln `open` (Windows) to the fresh child so its OPEN matches the frontend's dock.overlayOpen. Alternatively have the frontend re-issue setOverlay(true) after a codec switch that respawns the renderer.

### B9. Absolute pointer always lands on the PRIMARY monitor, never the streamed non-primary monitor
- **Area:** windows-input-inject  ·  **Platforms:** windows  ·  **Confidence:** high  ·  **Dims:** input-windows
- **Files:** desktop-app/crates/pulsar-core/src/input/windows.rs:166, desktop-app/crates/pulsar-core/src/input/windows.rs:169, desktop-app/src-tauri/src/host.rs:515
- **Scenario:** A multi-monitor Windows host. The client picks a non-primary monitor in the session menu (StreamReq.display_idx = 1, which is fully wired via host_displays() + CaptureHandle::switch_output). The client is a webview client (Windows-to-Windows webview, or macOS), which reads canvas coordinates and sends InputEvent::PointerMotion { x, y } normalized 0..1. The user moves the mouse over the streamed (second) monitor's video and clicks.
- **Symptom:** The host cursor jumps to and clicks on the PRIMARY monitor instead of the monitor being streamed. Pointer control of any non-primary monitor is completely broken for absolute-pointer (webview) clients; clicks/moves land on the wrong screen.
- **Root cause:** DesktopInput::pointer() computes dx = x*65535, dy = y*65535 and calls SendInput with MOUSEEVENTF_ABSOLUTE | MOUSEEVENTF_MOVE but WITHOUT MOUSEEVENTF_VIRTUALDESK. Per Win32, an absolute mouse move maps the 0..65535 normalized range to the PRIMARY display only unless MOUSEEVENTF_VIRTUALDESK is also set (in which case it maps to the whole virtual desktop). host.rs passes the client x,y straight through with no per-monitor offset/scale (line 515-518 `let (rx, ry) = (x, y); d.pointer(rx, ry)`). The code comment at windows.rs:165-166 still assumes "the primary display (what ddagrab output_idx=0 captures)", predating multi-monitor display_idx support. There is no use of GetSystemMetrics(SM_*VIRTUALSCREEN) or the selected monitor's rect anywhere in the input crate.
- **Suggested fix:** Pass the streamed monitor's rect (left/top/width/height in virtual-desktop pixels) into DesktopInput, and for absolute motion convert the normalized 0..1 into virtual-desktop normalized coords: vx = (mon.left + x*mon.width) ; map across the full virtual screen via SM_XVIRTUALSCREEN/SM_CXVIRTUALSCREEN to 0..65535 and OR in MOUSEEVENTF_VIRTUALDESK. The host already knows the chosen display (DisplayInfo / display_idx) so thread that monitor geometry into the on_input closure.

### B10. Linux host can't type any printable character from a Linux client (type_char is a no-op)
- **Area:** input-replay-linux  ·  **Platforms:** linux  ·  **Confidence:** high  ·  **Dims:** input-linux
- **Files:** desktop-app/crates/pulsar-core/src/input/uinput.rs:265, desktop-app/src-tauri/src/kbdhook/linux.rs:757, desktop-app/src-tauri/src/kbdhook/linux.rs:770, desktop-app/src-tauri/src/kbdhook/linux.rs:735, desktop-app/src-tauri/src/host.rs:528
- **Scenario:** A Linux machine (e.g. an Orange Pi appliance or any Linux desktop) connects as the CLIENT in remote-desktop mode to a Linux HOST and the user types text — a URL in a browser, a password, a chat message, a filename.
- **Symptom:** Nothing is typed on the host. Letters, digits and punctuation produce no characters at all. Only non-printable keys (Enter, arrows, F-keys) and Ctrl/Alt/Win shortcuts work, because those still travel as Key/VK. Remote-desktop text entry is effectively dead for Linux->Linux sessions.
- **Root cause:** The Linux client (kbdhook/linux.rs) resolves every printable key through xkb to a Unicode codepoint and forwards it as InputEvent::Char(c): when xkb_state is Some and no shortcut modifier is held it inserts the keycode into char_keys and sends ONLY Char (line 770-772), explicitly suppressing both the key-DOWN (the `else if !char_keys.contains` branch never fires, line 773) and the key-UP (line 735 `if !char_keys.remove(&code)`). build_xkb_state() succeeds on any normal X11 client (setxkbmap present), so the Char path is the default. On the host side host.rs:528 routes InputEvent::Char(c) to DesktopInput::type_char(c), which on Linux (uinput.rs:265) is a hard `{}` no-op. The doc comment at uinput.rs:262-264 assumes 'the Key (keycode) path still works', but that assumption is false because the Linux client never sends Key for printables — it sends Char and suppresses Key. So the character is silently dropped at the host. (Windows/macOS clients send only Key, and the webview client sends only Key, so this is specific to a Linux client driving a Linux host.)
- **Suggested fix:** Implement type_char on the Linux uinput backend instead of no-op: synthesize the codepoint via a temporary keymap remap of an unused keycode (the approach Sunshine/wlroots virtual-keyboard use), or — simplest reliable fallback — when type_char is called, look up the codepoint in a us-layout char->evdev table and emit the matching keycode+shift on the existing keyboard device so at least ASCII works. Alternatively have the Linux client NOT suppress the Key path (also send the raw Key) when the negotiated host is Linux, so the host's working key() path carries the character.

### B11. Revoking control ("Sadece izleme") mid-press leaves keys/mouse buttons stuck down on the host
- **Area:** input-hub-routing  ·  **Platforms:** all  ·  **Confidence:** high  ·  **Dims:** input-macos-hub
- **Files:** desktop-app/src-tauri/src/host.rs:488, desktop-app/src-tauri/src/host.rs:498, desktop-app/src-tauri/src/connections.rs:119, desktop-app/crates/pulsar-core/src/input/macos.rs:327, desktop-app/crates/pulsar-core/src/input/windows.rs:216
- **Scenario:** A client is controlling a host. The user presses-and-holds a key (e.g. Shift, or W in a game, or holds the left mouse button to drag-select) and, while it is still held, the host operator opens the Connections window and toggles "Sadece izleme" (view only) on that peer — or the same peer is briefly toggled view-only by any automation.
- **Symptom:** The key/button is stuck down on the host: the host behaves as if Shift/Ctrl is permanently held (every later locally-typed letter is capitalized / treated as a shortcut), or a runaway drag-select rectangle is left active, until the entire session is torn down. Re-enabling control does NOT clear it because the host injector still believes nothing is held.
- **Root cause:** The `on_input` closure in host.rs checks `view_only` and `return`s for EVERY event when set (host.rs:491-499), which drops the matching key-up / button-up that would normally clear the per-key/per-button held state and modifier mask inside the long-lived `DesktopInput` (held_keys/held_buttons/flags on macOS macos.rs:169-178, equivalents on Windows/Linux). Either expose a `DesktopInput::flush_held()` and call it from the closure when the gate first trips, or drop+recreate the injector on the view-only edge.

### B12. WASAPI loopback re-init on default-device switch desyncs ffmpeg's fixed input format → garbled/wrong-pitch audio for the rest of the session
- **Area:** audio-loopback  ·  **Platforms:** windows  ·  **Confidence:** high  ·  **Dims:** audio-host
- **Files:** desktop-app/src-tauri/src/host/handlers.rs:35, desktop-app/src-tauri/src/host/handlers.rs:55, desktop-app/crates/pulsar-core/src/audio/loopback.rs:271, desktop-app/crates/pulsar-core/src/audio/loopback.rs:331
- **Scenario:** Windows host is streaming system audio via WASAPI loopback (the default path, no virtual-audio device). Mid-session the user (or Windows, or a device hotplug) switches the default render endpoint to a device with a different mix format — e.g. from 48 kHz/2ch/float to 44.1 kHz, or to a 5.1 (6ch) HDMI output, or to a 16-bit endpoint. This is exactly the case the DeviceInvalidated recovery loop was built to survive.
- **Symptom:** The client's audio turns into garbled noise / wrong pitch / wrong channel mapping (and stays that way) for the remainder of the session, even though video and the stream itself keep working.
- **Root cause:** spawn_loopback_audio() calls loopback_format(pinned) ONCE (handlers.rs:35) and bakes -f/-ar/-ac into the ffmpeg command line (handlers.rs:55-60); those args are immutable for the life of that ffmpeg process. ### B69. [medium] MAX_RELAY_PAYLOAD (22400 B) caps reflection size but, with default-unlimited rate, still allows a ~16x-MTU full-rate reflector toward the session peer
- **Files:** relay/src/lib.rs:45, relay/src/lib.rs:277, relay/src/lib.rs:295, relay/src/lib.rs:113  ·  **Platforms:** all
- **Symptom:** The relay forwards every ~22KB datagram to the other peer at full line rate with no cap, because allow() short-circuits to true when limits are unlimited (relay/src/lib.rs:371). The documented intent is that sessions carry only a 'small control channel' (bulk A/V goes direct), but nothing enforces smallness per-rate — only the per-datagram size. A peer can be flooded with up to 22KB-per-datagram traffic it must receive and process, and the relay's 65536-byte recv buffer (:113) accepts the inbound side. This is a peer-targeted amplification/flood that is on by default.
- **Root cause:** The only unconditional guard is the per-datagram size check at :277 (drop if payload > MAX_RELAY_PAYLOAD). There is no unconditional per-session or per-source RATE limit; the rate/volume limiter (allow(), :295-298) is entirely opt-in and unlimited by default (limits.is_unlimited() -> true at :371). ### B71. [low] Auth token comparison is not constant-time

- **Symptom:** authed() at relay/src/lib.rs:363 compares d.token == token using the derived PartialEq on [u8;16] (pulsar-proto Token, lib.rs:92), which is a byte-wise comparison that short-circuits at the first mismatching byte. ### B72. [low] RK3588 codec/monitor switch: reopen request lost on a TOCTOU between take() and REOPEN clear
- **Files:** desktop-app/crates/pulsar-render/src/video.rs:195, desktop-app/crates/pulsar-render/src/video.rs:56  ·  **Platforms:** linux
- **Symptom:** The second switch is silently dropped: the renderer reopens on the first SDP and keeps decoding the wrong codec/monitor (e.g. green/mosaic 'Multi-layer HEVC' spam, or the previous monitor's geometry) until the next genuine EOF or another switch, looking like a stuck 'switching…' that never resolves to the right stream.
- **Root cause:** request_reopen() (video.rs:56-60) stores REOPEN_SDP then sets REOPEN=true. The decode loop does `next = REOPEN_SDP.lock().take()` (video.rs:195) and then unconditionally `REOPEN.store(false)` (video.rs:196). If a second request_reopen runs after the take() but before the store(false), it sets REOPEN_SDP=Some(new) and REOPEN=true, which the store(false) then clobbers back to false. The first SDP is used; the new SDP sits in REOPEN_SDP with REOPEN=false, so decode_once never sees the reopen flag and runs to completion on the stale codec. (Session-side debounce reduces but does not eliminate this — it is a genuine read-modify-write race on the two atomics/mutex.)
- **Suggested fix:** Make the reopen handoff atomic with respect to the flag: clear REOPEN only via a compare/swap tied to whether REOPEN_SDP was actually empty after take(), or re-check REOPEN_SDP after store(false) and loop again if it became Some. Simplest: take the SDP and set REOPEN=false under the same REOPEN_SDP mutex lock, and have request_reopen set both under that lock too.
