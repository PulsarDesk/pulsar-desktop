### C1 [high|REG] Operator manually closing the Allow/Deny approval window leaks OPEN_POPUP_COUNT, permanently disabling all future approval prompts (host becomes unconnectable for attended connects)
- files: auth.rs, lib.rs, +page.svelte | symptom: After the operator closes 3 approval popups via the window chrome, OPEN_POPUP_COUNT stays pinned at >= MAX_CONCURRENT_POPUPS (3) for the rest of the process. Each manual close also silently holds the just-dismissed client's connection attempt open for up to IDLE(60s)/MAX_TOTAL(120s) (the client UI stays on 'Connecting') because the pending request's oneshot is never resolved.
- **FIX [fixed]:** Fixed OPEN_POPUP_COUNT leak when operator closes approval popup via OS window chrome. Two changes: (1) in close_approval_window (auth.rs), made fetch_sub unconditional using fetch_update/saturating_sub instead of gating it inside `if let Some(win) = get_webview_window(...)` — the window is already destroyed when this runs after an OS-chrome close, so the old gate skipped the decrement permanently; (2) in the WindowEvent::Destroyed handler (lib.rs), added detection of approve-<id> window labels that immediately resolves the pending oneshot with false (deny), so the client receives a denial at once instead of hanging for 60-120 s until the IDLE/MAX_TOTAL deadline fires. ### C2 [high|REG] Allow/Deny popup counter (OPEN_POPUP_COUNT) leaks on every user/OS-closed or task-aborted approval window → after 3 leaks the host permanently auto-denies ALL incoming auth connections (self-DoS)
- files: auth.rs:230, auth.rs:266, auth.rs:275, auth.rs:278, auth.rs:322 | symptom: After 3 approval popups are closed by the user/OS (or 3 session tasks are aborted mid-race), OPEN_POPUP_COUNT is permanently stuck at >= MAX_CONCURRENT_POPUPS (3).
