### C1 [critical|REG] Windows client: kbdhook flush_held uses blocking_send and is called from async Tauri commands (set_overlay / kbd_capture_stop) → tokio-worker PANIC that poisons GLOBALS and permanently kills input capture
- platforms: windows | files: imp.rs:106, imp.rs:110, imp.rs:121, imp.rs:288, imp.rs:796
- symptom: The async Tauri command thread PANICS the instant flush_held tries to release the first held key/button. Because the panic unwinds while the `globals()` MutexGuard is held, the GLOBALS mutex is POISONED — every subsequent `globals().lock().unwrap()` in handle_key / handle_mouse / enable / disable / engage panics too, so the client's keyboard+mouse capture is dead for the entire rest of the app run (and the held key/button is left stuck on the host because flush never completed). The user must restart the app. With no held key at the moment of the call there is no panic, so it looks intermittent — it bites exactly when the user was holding something (the common case the flush exists to handle).
- **FIX [fixed]:** Fixed two call sites in kbdhook/imp.rs that called the blocking `flush_held` from paths reachable by async Tauri commands (tokio worker threads), where `blocking_send` panics unconditionally. Both `release_engage` (called from `set_overlay` → `kbdhook::release`) and `disable` (called from `kbd_capture_stop`) now use the existing non-blocking `flush_held_wh` + `send_flush_events` pattern instead: events are collected under the MutexGuard, the guard is dropped, then events are sent with `try_send`. The MutexGuard acquisition in both sites also gained `unwrap_or_else(|e| e.into_inner())` poison recovery so a prior stray panic cannot permanently brick capture. No new code was introduced — the fix reuses helpers already added for the combo disengage paths (C16). Compiles clean (`cargo check -p pulsar-tauri` passes with only pre-existing warnings).

### C7 [medium|REG] Resident-renderer re-park on a failed reconnect overwrites the global slot without evicting/reaping the existing resident — leaks an orphaned pulsar-render (process + live EGL context) on Linux/RK3588, reintroducing the documented orphan-pile-up stutter
- platforms: linux | files: play.rs:816, play.rs:965, play.rs:1306
- symptom: The failure path re-parks the just-reused child by writing `*state.resident_render.lock().unwrap() = Some(ResidentRender { … })` directly (play.rs:816 and play.rs:965), overwriting whatever was already in the Option. The previously-parked ResidentRender is dropped, and `std::process::Child::drop` on Unix neither kills nor waits the process — so that renderer is leaked: it keeps running with its EGL/GL context alive, its X window, and a bound RTP UDP socket, never reaped until the app exits. After several such events the Pi accumulates orphaned pulsar-render processes contending for the Mali GPU — the exact 'stutter after many reconnects / host stale' class MEMORY (opi5-input-stutter) says was cured. PR_SET_PDEATHSIG only reaps them on app exit, not between sessions.
- **FIX [fixed]:** The bug was already fixed in the current working tree as part of the uncommitted batch. The root cause was that `AppState::resident_render` was `Mutex<Option<ResidentRender>>` and all three park sites used `*lock = Some(...)` overwrite — both failed-reconnect re-park sites (request_launch and request_stream failure paths) and the normal stop_stream park site. Any pre-existing parked renderer had its `Child` silently dropped (not killed, not waited), leaking the process on Linux. The fix changed the field type to `Mutex<Vec<ResidentRender>>`, switched the single-take reuse path from `.take()` to `.pop()`, and replaced all three `= Some(...)` assignments with `.push(...)`. The old stop_stream eviction block (`take()` + `stop_render_child_blocking` on any prior occupant) was also removed — that was the most dangerous site because it actively SIGTERMed a concurrent tab's renderer. ### C8 [medium|REG] current_output()/cur_display report the requested monitor index even when find_output silently fell back to another output, so a hot-unplug of (or an out-of-range index for) the streamed monitor desyncs video vs absolute-pointer input on Windows
- platforms: windows | files: device.rs:294, device.rs:529, lib.rs:354, handlers.rs:1830, host.rs:888
- symptom: Video keeps flowing but on the WRONG monitor (the find_output fallback = first attached output, usually the primary), while absolute pointer moves/clicks land on neither the streamed-on-screen content nor a re-resolved monitor: cur_display/current_output() report the stale/out-of-range index, display_rect(that_index) returns None, so on_input never updates its MonitorRect and keeps mapping to the last-resolved (or primary-only) rect. The user sees the cursor/clicks land on a screen that doesn't match the video, with no recovery until an unrelated restream re-resolves geometry.
- **FIX [fixed]:** Fixed the three-layer desync between captured monitor and input mapping on Windows:

1. `find_output` now returns `(IDXGIAdapter1, IDXGIOutput1, u32)` — the third field is `actual_idx`, the real attached-output index chosen (same as `output_idx` when found; `0` when the fallback fires because the requested index is out of range).

2. `create()` shadows `output_idx` with `actual_idx` immediately after `find_output`, so `self.output_idx` is always the real captured monitor, never the stale/invalid requested index.

3. `build_duplication()` (the transient-retry re-enumeration path) updates `self.output_idx = actual_idx` after a successful `find_output` call, covering the race where an unplug happens during the retry loop.

4. `reinit()` (called from pacing.rs on ACCESS_LOST / mode change) likewise sets `self.output_idx = actual_idx` after `find_output`, so any subsequent `RunExit::Switch(self.output_idx)` carries the real output index, and the lib.rs outer loop's `cur_out_t.store(output_idx, ...)` publishes the correct monitor to `current_output()`.

5. `handlers.rs` now calls `pulsar_capture::list_displays()` to get the live display count and clamps `req.display_idx` to `[0, count-1]` before passing it to `start_capture_encode`, providing a cheap early guard for stale-picker scenarios. The `cur_display.store(...)` call after a successful init now uses `h.current_output()` (the thread-confirmed actual index) instead of `req.display_idx`, making it robust against the narrow window between the clamp and the thread's first build.

### C9 [medium|REG] Attacker-controlled OTP rotation cadence (note_global_failure → rotate_session_password) can deny a legitimate user access to their own host
- platforms: all | files: auth.rs:127, auth.rs:279, host.rs:452, host.rs:433, commands.rs:249

- **FIX [fixed]:** Added a 60-second `ROTATION_COOLDOWN` gate inside `note_global_failure()` in the `throttle` module. ### C10 [medium|REG] Host display-mode watcher re-baselines to the OLD monitor's size after a mid-confirm monitor switch, causing a spurious capture restart (stream blip) on the next poll

### C11 [medium|REG] SessionCleanupGuard kills ffmpeg encoder children but never wait()s them → one un-reaped zombie per encoder leaks on every go_online-during-session (abort path)

### C15 [medium|REG] Download concurrency guard is a no-op: api.fsGet resolves on request-enqueue, not transfer-completion, so all queued downloads fire at once and can still trip the reassembler's 8-transfer eviction cap (the exact failure C15 'fixed')
- platforms: all | files: Files.svelte:99, Files.svelte:107, Files.svelte:116, Files.svelte:118, session_cmds.rs:1138
- symptom: For DOWNLOADS the guard does nothing: more than 6 (indeed all) FsGet requests go out essentially simultaneously, so the host streams >8 files back concurrently and the client-side reassembler (hold.rs f_xfers) hits MAX_CONCURRENT_XFERS=8 and evicts the oldest active transfer when the 9th FileBegin arrives. The evicted download then fires file-recv{ok:false} → 'indirilemedi' even though nothing was wrong on the network — exactly the symptom C15 set out to eliminate. (Uploads are unaffected because their guard works; downloads regress.)
- **FIX [fixed]:** The download concurrency guard was a no-op because `api.fsGet()` resolves the moment the FsGet message is enqueued in the Rust channel (near-instant), not when the host finishes streaming the file back. `startXfer`'s `fn().finally()` therefore fired immediately, decrementing `activeXfers` before any data had been transferred, letting all queued downloads fire at once and trip the reassembler's 8-transfer eviction cap.

Fix: `download()` now awaits a second Promise after `api.fsGet()`. That Promise registers a `{resolve, reject}` entry in a component-scoped `pendingDownloads` Map (keyed by filename, FIFO per name) and only resolves when `onFileRecv` signals it — i.e., when the actual `file-recv` event for that filename arrives. The `onFileRecv` handler (inside the existing `$effect`) was extended to dequeue the first pending entry for the received filename and call `resolve()` or `reject()` based on `e.ok`. A 30-second `setTimeout` fallback removes the entry and calls `resolve()` so host-refused / jailed / lost-FileEnd transfers don't leak the slot permanently. `startXfer`'s `fn().finally()` now fires only after the complete async function (including the `file-recv` wait), making `activeXfers--` happen at real transfer completion — matching the upload path, which already awaited the full push.
