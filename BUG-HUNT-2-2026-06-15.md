### C2 [high] FileEnd changed from a unit variant to a struct variant — breaks cross-version file transfer (silent, transfer never completes)
- **REGRESSION** | platforms: all | dim: regress-wire-proto | files: desktop-app/crates/pulsar-core/src/service/wire.rs:233, desktop-app/crates/pulsar-core/src/service.rs:99, desktop-app/crates/pulsar-core/src/service/client.rs:84, desktop-app/src-tauri/src/host/handlers.rs:2060, desktop-app/src-tauri/src/play/hold.rs:367
- scenario: Two Pulsar builds that both report the same relay PROTOCOL_VERSION (still 2 — it was NOT bumped) but differ in the just-landed DataMsg shape connect and run a file transfer. Concretely: an Orange Pi appliance auto-updates to the new build (FileEnd-with-id) and connects to a not-yet-updated desktop host (old FileEnd unit), or vice versa. The user downloads a host file ('indir') or uploads one.
- symptom: The transfer never completes: the receiver buffers every FileChunk fine but silently drops the FileEnd, so the file is never flushed to 'Pulsar Alınanlar' / never saved, and the transfer hangs or is reported incomplete. No crash, no error log on the wire layer — just a dead transfer. Affects BOTH directions (new peer can't decode old's FileEnd; old peer can't decode new's FileEnd).
- root cause: The round-1 fix added a per-transfer `id` to the file-transfer messages and reshaped FileEnd from a UNIT variant (`FileEnd`) into a STRUCT variant (`FileEnd { #[serde(default)] id: u32 }`) at wire.rs:233-236. Service messages are serialized with serde_json (service.rs:95-101 `enc`/`dec`). In serde_json a unit variant serializes as the bare string `"FileEnd"`, while a struct variant serializes as `{"FileEnd":{"id":0}}` — these two encodings are mutually UN-decodable (empirically verified: `serde_json::from_str::<New>("\"FileEnd\"")` => Err `invalid type: unit variant, expected struct variant`; and the reverse => Err `invalid type: map, expected unit`). `#[serde(default)]` only papers over a MISSING field inside an already-struct variant — it does nothing for a unit→struct shape change. So the doc comment's claim that 'id defaults to 0 so an old peer that omits it still decodes' is correct for FileBegin/FileChunk (already struct variants → genuinely compatible) but FALSE for FileEnd. `dec()` returns None on the failed decode (client.rs:84 decode_data, host handlers), so the FileEnd is silently swallowed rather than panicking. PROTOCOL_VERSION was not bumped, so the relay's new version gate does not keep these peers apart — they fully connect.
- suggested fix: Either keep FileEnd a unit variant and carry the terminating id elsewhere (e.g. a separate `FileEndId(u32)` appended variant, with plain `FileEnd` retained for old peers), OR bump PROTOCOL_VERSION so mismatched builds refuse to register and surface the existing 'update required' error instead of silently breaking transfers. If FileEnd must become a struct variant, add a custom Deserialize that accepts both the bare-string and the map form (id defaulting to 0) so old↔new file transfers still complete. Add a wire test that decodes the literal old-form bytes (`"FileEnd"`), not just a fresh roundtrip — the existing fs_*/file roundtrip tests all encode-then-decode the SAME build, so they can never catch this.
- verifier correction: Minor framing refinements (do not change the verdict): (a) The relay's IncompatibleVersion gate uses bincode-serialized relay control messages and a version-number compare; DataMsg side-channel messages are serde_json over the encrypted session, so even setting aside the no-bump issue, the version gate is the wrong layer to catch a same-version wire-shape drift. (b) The suggested fix's cleanest option is a custom Deserialize on FileEnd accepting both the bare-string and {"id":..} forms (plus the symmetric concern: an OLD peer can never be taught to accept the new map form, so to support old<-new you must also keep emitting the bare string to old peers — which is unknowable per-peer). Practically, the robust fix is to add a NEW appended variant (e.g. FileEndV2 { id }) and keep emitting plain `FileEnd` for back-compat, OR bump PROTOCOL_VERSION so mismatched builds refuse to pair. (c) Add a wire test that decodes the LITERAL old bytes (the string "FileEnd"), since all existing roundtrip tests encode+decode the same build and cannot catch this.
- **FIX [fixed]:** FileEnd unit→struct variant shape change now accepts both wire forms. A serde_json::Value normalization step intercepts the legacy bare-string "FileEnd" (old unit-variant form) and rewrites it to {"FileEnd":{}} before the typed DataMsgWire deserializer sees it. All 14 wire tests pass including 4 new FileEnd-specific regression tests that decode literal old-form bytes.
  - follow-up: The serde_json::Value intermediate adds one extra allocation per DataMsg decode. For file-transfer control messages this is negligible. If performance matters long-term, a proper serde Deserializer-level fix would require forking serde_json or implementing a two-pass streaming parser — not worth it here. PROTOCOL_VERSION was not bumped (per the bug report scenario), so mixed-version peers can still connect and now correctly exchange FileEnd in both directions.
  - xplat risk: None. The fix is in a pure serde/serde_json layer with no platform-conditional code. DataMsg is used identically on Windows, macOS, and Linux.
