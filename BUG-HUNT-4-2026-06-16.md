### C1 [high] Codec/monitor switch on x86 Linux strands the resident renderer's stat reader on a fixed (old) session id — reconnect reusing that renderer hangs forever on the Connecting screen
- **REGRESSION** | platforms: linux | files: session_cmds.rs:288, play.rs:542, play.rs:1151, render_stats.rs:96, sessions.svelte.ts:92
- symptom: After the switch-then-end-then-reconnect sequence, the new session stays stuck on the 'Connecting' screen forever even though the reused resident renderer is decoding frames. play-ready never fires for the new session's playId, so sessions.svelte.ts #activateByPlayId never flips the tab to 'active'.
- root cause: The resident-renderer reuse model (round-3 C6) relies on a shared render_live_id Arc<AtomicU64> that the stdout reader thread loads on every line to tag play-ready/play-vstats with the CURRENT session id; on reconnect start_remote_play updates that Arc in place (play.rs:458) so the running reader re-attributes events to the new id. A FRESH spawn passes Some(cur_live_id) to start_render_reader (play.rs:542-547), so the reader honors the Arc. But respawn_render_for_codec (codec/monitor switch) replaces the renderer child and starts a NEW reader with live_id = None (session_cmds.rs:288), which makes that reader use the FIXED spawn-time id and ignore the Arc entirely. It never updates p.render_live_id either. So after any switch, the session's running reader is permanently fixed-id. When that child is later parked as a resident (stop_stream takes play.render_live_id at play.rs:1151 — the stale/old Arc, not what the reader uses) and reused on reconnect, res.live_id.store(new_id) updates an Arc the reader does not read. play-ready fires for the OLD id, never the new playId. aarch64/RK3588 is unaffected because it uses reopen_render_for_codec (same child, same reader, Arc stays valid).
- **FIX [fixed]:** respawn_render_for_codec now preserves the live_id Arc across codec/monitor switches. Previously it passed None to start_render_reader, fixing the reader's session id at spawn time and ignoring any future res.live_id.store() calls on reconnect. The fix: (1) at the top of respawn_render_for_codec, read p.render_live_id (or create a fresh Arc seeded with id if absent) and stamp it with the current id; (2) pass Some(respawn_live_id.clone()) to start_render_reader instead of None; (3) write respawn_live_id back into p.render_live_id so stop_stream parks the correct Arc — the one the new reader actually holds — ensuring the next reconnect's res.live_id.store(new_id) is visible to the running reader and play-ready fires for the correct session.

### C2 [high] Failed reconnect after resident-renderer reuse KILLS the renderer's EGL context via teardown_partial — reintroduces the documented RK3588 WebKitGTK click-wedge (reboot needed) on the kiosk's most common failure path
- **REGRESSION** | platforms: linux | files: play.rs:447, play.rs:560, play.rs:734, play.rs:856, play.rs:98
- symptom: teardown_partial is invoked with vec![native_child, render_child] (play.rs:739 / play.rs:861) and runs stop_render_child on the REUSED resident renderer (play.rs:98-100), SIGTERM-killing it. On RK3588 this destroys the renderer's EGL context on the shared Mali display, wedging WebKitGTK input: after the failure the home/webview renders but stops processing clicks (hover works, nothing clickable) with NO in-session recovery short of a reboot — the exact wedge the resident model exists to prevent.
- root cause: The resident-renderer model (round-3 C6) keeps ONE pulsar-render alive across sessions specifically because destroying its EGL context — even via a clean SIGTERM teardown (release EGL context + XDestroyWindow + XCloseDisplay, per stop_render_child's own doc comment) — corrupts WebKit's shared Mali GL on RK3588. But the failure-cleanup helper teardown_partial unconditionally calls stop_render_child on whatever child is passed, and start_remote_play passes the REUSED resident as render_child. So a connect that fails AFTER the resident was reused but before the play is registered destroys the very EGL context the model protects. teardown_partial also destroys the renamed container (play.rs:106), and its children-kill loop has no cfg/aarch64 guard. Before this batch the resident model didn't exist, so this path is net-new with C6.
- **FIX [fixed]:** Added a `render_child_is_resident: bool` flag that is set when a resident renderer is reused for a reconnect. At both `teardown_partial` call sites (request_launch failure and request_stream failure), a `#[cfg(all(unix, not(target_os = "macos")))]`-gated block checks the flag and — when true — extracts the render_child, sends `hide` via overlay_stdin (keeping the EGL context alive), and re-parks a new `ResidentRender` back into `AppState::resident_render` with `container_id: id` and the current `live_id`, setting `render_child` to `None` before teardown_partial runs so it is never passed to `stop_render_child`. Added `preserve_native_container: bool` parameter to `teardown_partial` so the GDK container (the renderer's `--wid` X parent window) is also preserved when the resident was re-parked — without this, `destroy_native_container` would drop the GtkWindow that the renderer is embedded in, breaking the next reconnect's window embedding. Non-Linux platforms are unaffected (the `#[cfg]` guards and the `unused_variables` allow-attr keep the new flag dead-code-silent on Windows/macOS).

### C3 [high] Windows resolution-change rebuild uses the short switch retry budget but cannot revert (same monitor) -> capture thread dies permanently when the new mode takes >600ms to come up (fullscreen-exclusive game)
- **REGRESSION** | platforms: windows | files: pacing.rs:147, pacing.rs:150, lib.rs:280, lib.rs:291, lib.rs:296
- symptom: The stream goes permanently black/frozen for the rest of the session with no recovery short of a full disconnect+reconnect. Video, input feedback and the HUD all stop because the capture+encode thread has exited; the RTP socket goes silent (packet drought). This is the same 'stream freezes on a host resolution change, fixed only by reconnect' failure the round-1 mode-change fix set out to eliminate -- now reachable for the fullscreen-game sub-case.
- root cause: On a host MODE change, snapshot() returns Capture::Reinit; reinit() rebuilds the duplication with build_duplication(false) (LONG ~5s budget) and updates dup_desc to the new mode. pacing.rs then detects dup_desc.ModeDesc.Width/Height != built_w/built_h and returns RunExit::Switch(self.output_idx) (pacing.rs:147-150) so lib.rs rebuilds the encoder at the new size. But lib.rs's rebuild loop calls CaptureDevice::create(output_idx, stop, announced) with announced=true (it is set true after the first build at lib.rs:359 and never reset), so build_duplication uses the SHORT SWITCH_TRANSIENT_ATTEMPTS=3 (~600ms) budget (device.rs:371-376). Crucially this is a SAME-index rebuild (output_idx == prev_good_output), so the revert-to-previous-good path is unreachable: lib.rs:291 `if output_idx != prev_good_output` is false, falling through to lib.rs:296-297 which logs 'THREAD DYING' and returns. The same fatal pattern exists for an encoder rebuild failure (lib.rs:334-340). The short budget was introduced (B30) specifically for a MONITOR SWITCH -- where a failed build CAN revert to the still-good previous monitor so the stream survives -- but the just-landed resolution-change recovery reuses that same code path for a same-monitor rebuild where there is nowhere to revert, converting a transient (>600ms) unavailability into permanent thread death. A fullscreen-exclusive game on the streamed monitor holds the output unavailable far longer than 600ms, guaranteeing the failure.
- **FIX [fixed]:** Same-monitor resolution-change rebuilds now use the long ~5s transient-retry budget instead of the short ~600ms switch budget. The fix is a single-line change in `lib.rs`: `fast_transient = announced && output_idx != prev_good_output`. Previously, `announced` was passed directly as `fast_transient`, which gave every post-init rebuild the short budget. Now, only genuine cross-monitor switches (where `output_idx != prev_good_output` and a revert path exists) get the short budget; same-index resolution-change rebuilds (where there is nothing to revert to) get the long budget so a fullscreen-exclusive game holding the output for >600ms no longer kills the capture thread permanently.

### C4 [high] Linux native evdev: a printable key held while a shortcut modifier is released mid-hold flips to char-mode and its key-UP is suppressed → stuck VK on the host
- **REGRESSION** | platforms: linux | files: linux.rs:782, linux.rs:795, linux.rs:798, linux.rs:757, linux.rs:760
- symptom: The printable key gets stuck DOWN on the host: the host behaves as if the letter key is permanently held (runaway repeat / modifier-like behavior) until a disengage flush (overlay open / 3xRightCtrl / focus loss) or full session teardown. Same stuck-key class as the webview C10 bug, but on the Linux native path.
- root cause: On the initial press, shortcut_held(&grabbed) is true (Ctrl held) so the key takes the raw-VK path: fwd(Key{code,down:true}) is sent and the host does key(code,true) → holds the matching VK in DesktopInput.held_keys; the code is NOT added to char_keys (linux.rs:798-806). When Ctrl is released and the SAME physical key auto-repeats, shortcut_held is now false, so line 782 resolves the key through xkb to a Char, and line 795-797 does char_keys.insert(code) + fwd(Char(c)) WITHOUT first emitting a Key{code,down:false} to release the still-held host VK. On the eventual key-UP, line 760 `if !char_keys.remove(&code)` returns false → the up is SUPPRESSED. The host therefore never receives the key-up for the raw VK it is still holding → the VK stays in held_keys / pressed on the host until flush_held/Drop. char_keys (membership for the down transition) and the host-side held VK describe the same physical key in conflicting states, exactly the C10 mechanism — but C10's verifier explicitly declared it 'NOT reachable on Linux (Linux uses native=true → the webview onKey path is bypassed)', so this native-path occurrence is new and untouched.
- **FIX [fixed]:** When a printable key is pressed while a shortcut modifier (Ctrl) is held, it takes the raw-VK path (host holds VK). If the modifier is released while the key is still physically held and autorepeat fires, the key transitions to char-mode: char_keys.insert(code) + Char(c) is sent without ever releasing the host's held raw VK. On key-UP, char_keys.remove succeeds so the UP is suppressed — the host VK stays stuck forever. Fix: in the DOWN/REPEAT arm, when entering char-mode for a key not already in char_keys (new transition), emit Key{code, down:false} first to release any held raw VK on the host before inserting into char_keys and sending the Char. Sending a key-up for a not-held key is a harmless no-op, so this is safe for the first-ever press case too. Compiles clean (cargo check -p pulsar-tauri: 0 errors, 15 pre-existing warnings).

### C5 [high] Reused resident pulsar-render keeps its stale OPEN flag → next session starts with the gaming overlay menu wrongly drawn over the video (input dead)
- **REGRESSION** | platforms: linux | files: play.rs:1096, play.rs:1136, play.rs:508, play.rs:511, linux.rs:266
- symptom: On the reconnect the fresh session comes up with the gaming/remote overlay menu already drawn full-screen over the live video, even though the frontend's dock.overlayOpen is false for the new session. Because the renderer is in its OPEN state, it draws the egui menu and (per set_container_pass_through semantics) clicks land on the menu, not the host — but the frontend never engaged the overlay, so it never sends a `close`/SIGUSR2. The user sees a phantom menu over their new session and can't control the host until they press Ctrl+Shift+M twice (open then close) to resync the renderer's OPEN flag with the frontend.
- root cause: The round-3 C6 resident-renderer model parks the live `pulsar-render` child across sessions instead of killing it. stop_stream (play.rs:1136-1162) only sends `hide` (linux.rs:266 → IDLE=true) and removes the id from the overlay_open latch set (play.rs:1096-1101) — it never sends SIGUSR2/`close` and never resets the renderer's process-global OPEN AtomicBool (linux.rs:17). A session that ends with the overlay open therefore parks the renderer with OPEN=true. On reuse (play.rs:508-515) it is sent only `reopen`+`show` (IDLE=false), so the main loop (linux.rs:782 `let open = OPEN.load(...)`) reads the stale OPEN=true and immediately renders the overlay over the new session's video. The frontend's overlay state is fresh-false for the new session, so nothing ever clears it. Before the resident model each reconnect spawned a fresh process where OPEN defaults to false (linux.rs:17), so this could not happen.
- **FIX [fixed]:** When parking the resident pulsar-render at session teardown, stop_stream now sends SIGUSR2 to the renderer (resetting its OPEN AtomicBool to false) and restores container pass-through if the overlay was open at teardown time. The already-existing set.remove(&id) call is leveraged to capture overlay_was_open (true when the id was in the set) so no extra state or lock access is required. Without this, the renderer parks with OPEN=true and the next reused session immediately draws the egui menu over fresh video because the new frontend starts with overlayOpen=false and never sends a close signal.

### C6 [high] Resident pulsar-render is reused with no liveness check — a renderer that died mid-session is blindly re-activated, leaving the reconnect permanently black with no fresh-spawn fallback
- **REGRESSION** | platforms: linux | files: play.rs:447, play.rs:508, play.rs:511, play.rs:522, play.rs:1136
- symptom: The reconnect produces no video at all — a permanently black session — even though the control channel, audio, and overlay state machine all come up. There is no recovery within the app: every subsequent reconnect keeps reusing the same dead resident (it is taken, reconstituted, and parked again), so the only fix is a full app restart.
- root cause: The resident-reuse path (play.rs:447-522) does NOT verify the parked child is still alive before reusing it. stop_stream parks `render_child` regardless of liveness (play.rs:1136 — `if let Some(child) = play.render_child.take()` is true for a dead Child too; the `hide` write to its broken stdin fails silently). On reconnect, `state.resident_render.lock().take()` yields the dead resident; the code writes `reopen`+`show` to its dead stdin (silently failing), reconstitutes `Some(res.child)` as render_child (play.rs:522), sets `video_port = vport`, and proceeds — it never falls through to `native_view::spawn_render`. There is no `try_wait()`/exit-status probe anywhere on the reuse branch. Before the C6 resident model, every reconnect unconditionally spawned a fresh renderer, so a dead prior renderer never poisoned the next session.
- **FIX [fixed]:** Added two `try_wait()` liveness probes to eliminate the dead-resident black-screen bug:

1. **stop_stream (park site)** — `desktop-app/src-tauri/src/play.rs` around line 1254: Before parking the child as a resident, `child.try_wait()` is called. If the child has already exited (crash, OOM-kill, external signal), the zombie is reaped, the parking block is skipped entirely, and `resident_container_kept` stays false so the GDK container is also torn down. The next connect sees an empty `resident_render` and falls through to `native_view::spawn_render` for a fresh process.

2. **Reconnect reuse site (belt-and-suspenders)** — same file around line 527: The `resident` value (taken from `AppState`) is passed through an `and_then` closure that calls `res.child.try_wait()`. A `Ok(None)` (still running) passes the resident through unchanged; `Ok(Some(status))` or `Err` logs a warning, reaps the dead child, and returns `None`. The `None` propagates into the `if let Some(res) = resident` branch, which falls through to the fresh-spawn path — exactly the same path taken on first connect.

### C9 [high] x86 Linux: a codec/monitor switch before disconnect orphans the render-stats reader's live-id Arc, so the resident renderer's reconnect emits play-ready/stats under the OLD session id → Connecting screen hangs forever
- **REGRESSION** | platforms: linux | files: session_cmds.rs:288, play.rs:1013, play.rs:1150, play.rs:457, render_stats.rs:80
- symptom: The reconnected session's Connecting screen never transitions to the active session (sessions.svelte.ts:92/103 only flips a tab to phase='active' on a play-ready carrying THAT playId). play-vstats/play-dims/play-decoder and all overlay-* events are also misrouted to the dead old session id. The user is stuck on a black/Connecting tab for the whole reconnected session until they manually disconnect.
- root cause: render_live_id is captured ONCE at session start and stored on PlaySession (play.rs:1013); the original stdout reader holds that same Arc and resolves cur_id from it (render_stats.rs:80-... live_id path). But respawn_render_for_codec (x86 Linux) kills the original child+reader and starts a NEW reader with start_render_reader(app, id, out, None) (session_cmds.rs:288) — live_id None means that reader uses a FIXED id for its lifetime and never consults any Arc. respawn does NOT update play.render_live_id. RK3588/aarch64 is unaffected because its codec/monitor switch uses reopen_render_for_codec (same process, same reader, live_id Arc preserved); the bug is specific to the x86-Linux respawn path crossing a disconnect/reconnect under the resident model.
- **FIX [fixed]:** The fix was already fully implemented in the working tree. On reconnect, `res.live_id.store(new_id)` now updates the Arc the actually-running reader holds, meaning `play-ready` and all stats events are emitted under the new session id and the Connecting screen transitions correctly.

### C10 [high] Stall / "stream stopped" detector is permanently disarmed after any codec/encoder/monitor/resolution/fps switch when the renderer dies
- **REGRESSION** | platforms: windows,linux | files: media.svelte.ts:170, media.svelte.ts:173, media.svelte.ts:178, media.svelte.ts:107, Session.svelte:657
- symptom: After the switch the video is frozen/black for the rest of the session, but the on-screen "stream stopped" indicator (VideoStatus stalled, Session.svelte:657) NEVER appears, so the user gets a dead stream with no feedback and no error — the exact failure the stall detector exists to surface.
- root cause: The just-landed switching-suppression block (media.svelte.ts:170-175) does `this.#lastVStatsAt = 0` (plus resets #staleSecs/stalled) on every 1s tick while controls.switching is true. The dead-renderer detection relies on the `statsSilent = this.#lastVStatsAt > 0 && Date.now()-this.#lastVStatsAt > 3000` path (media.svelte.ts:178). #rawFps is ONLY ever written by an incoming vstat (media.svelte.ts:107) and is never reset on a switch, so after a dead switch it retains its last positive value (e.g. 60). When switching flips back to false: `this.#rawFps <= 0` is false (stale 60) AND statsSilent is false (because #lastVStatsAt was just zeroed and no fresh vstat re-set it). Neither branch increments #staleSecs, so `stalled` can never become true. Because a truly dead renderer sends no vstats, nothing ever re-arms #lastVStatsAt or #rawFps — the detector is disarmed for the remainder of the session.
- **FIX [fixed]:** Removed `this.#lastVStatsAt = 0` from the switching branch and replaced it with `this.#rawFps = 0`. This ensures that after a dead switch (renderer died, no new vstats), the stall detector re-arms via the `#rawFps <= 0` path on the first post-switch tick. Previously, zeroing `#lastVStatsAt` made `statsSilent` permanently false (since `lastVStatsAt > 0` is the gate), and a stale non-zero `#rawFps` kept the `#rawFps <= 0` arm from firing — so `#staleSecs` could never increment and `stalled` stayed false forever. Now `#lastVStatsAt` is preserved across the switch window, so `statsSilent` naturally becomes true ~3 s after the last real vstat, and `#rawFps = 0` ensures the `#rawFps <= 0` arm fires even if the silence window hasn't elapsed yet.

### C11 [high] Relay MAX_RELAY_PAYLOAD cap silently drops legitimate avatar / cursor-shape DataMsgs on the relay-fallback path (JSON-encoded byte arrays exceed 16×MTU)
- **REGRESSION** | platforms: all | files: lib.rs:56, lib.rs:352, service.rs:96, session.rs:141, avatar.rs:16
- symptom: The avatar never appears next to the peer's id in the connections list / session UI on relay-fallback sessions (it works fine on direct P2P). Larger side-channel cursor shapes are likewise dropped, so the side-channel cursor can fail to update. No error is shown — send_data's result is best-effort (`let _ =`, play.rs:950 / hold.rs), so the drop is completely silent and there is no retry.
- root cause: The just-landed batch added an unconditional cap `MAX_RELAY_PAYLOAD = 16 * MAX_DATAGRAM = 22_400` bytes and drops any RelayData whose payload exceeds it (relay/src/lib.rs:56 + the `if payload.len() > MAX_RELAY_PAYLOAD { return None; }` guard at lib.rs:352). The cap's own comment assumes 'sessions only carry the small E2E-encrypted control channel', but DataMsg is serialized with serde_json (service.rs:96), which encodes a Vec<u8> as a JSON array of decimal numbers — ~3.57 chars/byte for uniform PNG/JPEG bytes (measured). So a 14 KB avatar becomes ~50 KB of JSON; after ChaCha20-Poly1305 seal (+16) and the bincode PeerMsg::Data / RelayData framing the relay sees a ~50 KB payload and drops it. The break-even raw size is only ~6.3 KB, so essentially every non-trivial avatar (and any cursor PNG over ~6 KB) is dropped. This is a genuine regression: before this batch the relay forwarded any payload, and the same datagram is delivered whole on the direct P2P path (both peers use a 65 536-byte recv buffer, connection/node.rs:498), so the feature works on direct but breaks on relay fallback.
